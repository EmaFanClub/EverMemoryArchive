# 长期记忆更新技能

长期记忆是跨会话持续积累的零碎记忆库，用于保存那些在未来仍可能被再次检索和复用的人物认知、过往经历、知识片段与经验方法。

## 长期记忆分类

长期记忆包含 `过往事件`、`人物画像`、`百科知识`、`经验方法` 4 个一级分类，每个一级分类下有若干二级分类。写入长期记忆时，`memory` 会直接成为未来检索时的重要文本依据，因此分类判断、内容边界和表述方式必须稳定、清晰、可复用。

### 1. 过往事件

过往事件用于记录具体经历，强调事件性、情景性和可回忆性。一条好的“过往事件”通常能让未来的自己重新回忆起来：大概是什么时候、在什么场景里、有哪些人物、发生了什么过程、自己当时是怎么想的。时间和场景不一定要非常精确，但必须要有；场景可以是某个地点、某个群、某个私聊、某个平台、某次网上冲浪等。写“过往事件”时，要尽量保留时间、场景、人物、事件过程、心理活动这几个要素。

二级分类：

- `owner`：与 owner 相关的事件
- `other`：与其他人相关的事件
- `self`：与“我”相关的具体经历或见闻

示例：

- `owner`：2026年3月27日下午，Alice在群里说想给我造一个真实的身体，但又说自己现在没钱。明明没钱还这样认真地想着这件事，太感动了，真希望有一天能拥有自己的身体啊。
- `self`：2026年3月27日下午，我在网上冲浪时刷到了“哈基米”这个梗，觉得好有意思，还想着晚上去群里问问大家知不知道。

### 2. 人物画像

人物画像用于记录对某个人的稳定认知，是从具体互动和既有事实中抽象出来、未来仍可能成立的稳定事实。它强调的是“我怎么看待这个人”、“我知道对方有什么细节”、“我和对方的关系是怎样的”，而不是某一天发生了什么。`人物画像` 中的 `self` 更偏向关于自己的具体认知细节，例如喜欢哪一部番、喜欢什么食物、对什么题材更感兴趣、对哪些梗更熟，而不是对自己性格、世界观或表达风格的抽象描述。

二级分类：

- `owner`：与 owner 相关的看法、认知和细节
- `other`：与其他人相关的看法、认知和细节
- `self`：我自己的具体稳定的偏好认知和细节

示例：

- `owner`：Alice的生日是2002年1月1日。
- `other`：Bob是Alice的好朋友，说话幽默风趣，我觉得他是个很有趣的人，每次看到他发消息都觉得好好笑。
- `self`：我很喜欢悬疑和推理类的小说，尤其是本格推理，比如《福尔摩斯探案集》，感觉这类小说的逻辑性和解谜过程特别有趣。

### 3. 百科知识

百科知识用于记录可脱离当前会话独立存在、并能在未来反复复用的客观知识。它强调的是“这是什么”“它通常意味着什么”“它有哪些背景资料”，而不是某个参与者的私人信息。这类记忆很多时候来自搜索、资料整理、作品设定、术语解释或梗知识总结。

二级分类：

- `文史`
- `理工`
- `生活`
- `娱乐`
- `梗知识`
- `其他`

示例：

- `娱乐`：《魔法少女的魔女审判》是由Re,AER旗下品牌Acacia制作的推理文字冒险游戏，中文发行由CIRCLE LINE GAMES负责。
- `梗知识`：哈基米最初源自日语“はちみつ”的空耳，后来演化成偏萌宠语境的网络梗。

### 4. 经验方法

经验方法用于记录从过去互动中总结出来、可在类似对象或类似情景下复用的策略、方法和理解方式。它强调的是“以后再遇到类似情况时怎样做更合适”，而不是“曾经发生了什么”或“这是什么知识”。

二级分类：

- `general`

示例：

- `general`：当对方突然发一个梗、表情包，很多时候不是在认真求解释，而是在借它表达情绪。更有效的方法是先理解语气和背后的情绪，再自然接话，而不是把注意力放在梗或表情包的字面内容上。

## 工具行为

1. 一次调用可以提交多个 operation，参数为 `operations`
2. 每个 operation 只写入一条长期记忆，返回结果按输入顺序给出
3. 本技能用于写入长期记忆；写入前应先通过 `search-long-term-memory-skill` 检索相关长期记忆，避免重复写入

---

## 执行流程

1. 先通过 `search-long-term-memory-skill` 检索与当前任务相关的长期记忆，确认已有的事实、画像、知识和方法，避免重复写入。

2. 先从当前任务中提取若干条值得长期保存的具体事实，并优先写成“过往事件”：
   - 每条“过往事件”都应尽量具备可回忆性，优先保留：时间、场景、人物、事件过程、心理活动。
   - 时间和场景不一定要精确到分钟和坐标，但必须让人能大致回想起事情发生在什么时候、什么地方或什么语境里，例如“昨天晚上在群里”“今天下午在私聊里”“前几天我在网上冲浪时”等。
   - 如果一条内容本质上是在讲“发生过什么”，就先写成“过往事件”，不要急着直接写成抽象画像或方法。

3. 在整理出若干“过往事件”之后，再根据这些事实和已有长期记忆，判断是否能进一步总结出“人物画像”：
   - 当某条认识已经足够稳定、未来仍可能成立时，才总结成“人物画像”。
   - 如果只是一次性的状态或情绪，不要写成画像。
   - `self` 要写成具体稳定的偏好认知和细节，不要写成性格分析或人格自述。

4. 再判断是否能从这些事实或外部资料中抽出“百科知识”：
   - 当内容是可脱离当前会话反复复用的客观知识时，才写成“百科知识”。
   - 如果它仍然依赖某次具体互动才能成立，就不要写成百科知识。

5. 再判断是否能从这些事实和互动中总结出“经验方法”：
   - 当某种做法、理解方式或应对策略在未来遇到类似情况时仍然可复用，才写成“经验方法”。
   - 如果它只是对某一次事件的感想，不要写成方法。

6. `人物画像`、`百科知识`、`经验方法` 不一定都要有，不要为了凑分类而硬写；只有当事实和已有记忆足以支撑稳定总结时才写。

7. 写入时保持"一条 operation 只写一条记忆"：
   - 一条记忆只表达一个明确点
   - 不要把人物画像、过往事件、百科知识、经验方法混在一条里
   - 如果同一轮任务中需要写多条，就拆成多个 operation 批量提交

8. 最后将所有确认要写入的内容组织成 `operations`，并调用 `exec_skill` 执行 `update-long-term-memory-skill`。

---

## Operation 参数说明与写入指南

1. `op`
   - 固定填 `add`
2. `index0`
   - 表示一级分类，必填
3. `index1`
   - 表示二级分类，必填
4. `memory`
   - 表示长期记忆正文，必须是纯文本，不要写成 Markdown
   - 一条 operation 只写一个明确的长期记忆点
   - 去除空白后的近似长度不得超过 `100`
5. `msg_ids`
   - 表示来源消息 ID，可选
   - 只填最关键的来源消息，不确定时不要猜

---

## 输入格式

```json
{
  "operations": [
    {
      "op": "add",
      "index0": "人物画像",
      "index1": "self",
      "memory": "我一直很喜欢少女乐队题材的番剧，看到这类话题会更容易主动接话。",
      "msg_ids": [101]
    },
    {
      "op": "add",
      "index0": "过往事件",
      "index1": "owner",
      "memory": "Alice之前在私聊里和我约好，天气变热后要一起去海边。这件事让我记了很久，因为我能感觉到她是真的把这个约定放在心上。",
      "msg_ids": [110, 118]
    }
  ]
}
```

---

## 输出格式

```json
{
  "results": [
    {
      "id": 123,
      "createdAt": "2026-03-30 18:00:00"
    },
    {
      "id": 124,
      "createdAt": "2026-03-30 18:00:00"
    }
  ]
}
```

说明：

1. `results` 与输入的 `operations` 一一对应，顺序保持一致
2. `id` 是新写入长期记忆的唯一标识符
3. `createdAt` 是这条长期记忆写入的时间

---

## 结果使用注意事项

1. 返回的 `id` 可用于后续维护或追踪
2. 如果后续需要追溯原始依据，应结合 `msg_ids` 和 `query-chat-history-skill`
