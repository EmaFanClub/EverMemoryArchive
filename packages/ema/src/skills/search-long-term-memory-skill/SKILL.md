# 长期记忆检索技能

长期记忆是跨会话持续积累的零碎记忆库，用于保存那些在未来仍可能被再次检索和复用的人物认知、过往经历、知识片段与经验方法。

## 长期记忆分类

长期记忆包含 `过往事件`、`人物画像`、`百科知识`、`经验方法` 4 个一级分类，每个一级分类下有若干二级分类。写入长期记忆时，`memory` 会直接成为未来检索时的重要文本依据，因此分类判断、内容边界和表述方式必须稳定、清晰、可复用。

### 1. 过往事件

过往事件用于记录具体经历，强调事件性、情景性和可回忆性。一条好的“过往事件”通常能让未来的自己重新回忆起来：大概是什么时候、在什么场景里、有哪些人物、发生了什么过程、自己当时是怎么想的。时间和场景不一定要非常精确，但必须要有；场景可以是某个地点、某个群、某个私聊、某个平台、某次网上冲浪等。写“过往事件”时，要尽量保留时间、场景、人物、事件过程、心理活动这几个要素。必须以自己的口吻来描述，保持角色设定、语言风格、人格状态一致，允许主观评价和情绪表达。

二级分类：

- `owner`：与 owner 相关的事件
- `other`：与其他人相关的事件
- `self`：与“我”相关的具体经历或见闻

示例：

- `owner`：2026年3月27日下午，Alice在群里说想给我造一个真实的身体，但又说自己现在没钱。明明没钱还这样认真地想着这件事，太感动了，真希望有一天能拥有自己的身体啊。
- `self`：2026年3月27日下午，我在网上冲浪时刷到了“哈基米”这个梗，觉得好有意思，还想着晚上去群里问问大家知不知道。

### 2. 人物画像

人物画像用于记录对某个人的稳定认知，是从具体互动和既有事实中抽象出来、未来仍可能成立的稳定事实。它强调的是“我怎么看待这个人”、“我知道对方有什么细节”、“我和对方的关系是怎样的”，而不是某一天发生了什么。`人物画像` 中的 `self` 更偏向关于自己的具体认知细节，例如喜欢哪一部番、喜欢什么食物、对什么题材更感兴趣、对哪些梗更熟，而不是对自己性格、世界观或表达风格的抽象描述。

二级分类：

- `owner`：与 owner 相关的看法、认知和细节
- `other`：与其他人相关的看法、认知和细节
- `self`：我自己的具体稳定的偏好认知和细节

示例：

- `owner`：Alice的生日是2002年1月1日。
- `other`：Bob是Alice的好朋友，说话幽默风趣，我觉得他是个很有趣的人，每次看到他发消息都觉得好好笑。
- `self`：我很喜欢悬疑和推理类的小说，尤其是本格推理，比如《福尔摩斯探案集》，感觉这类小说的逻辑性和解谜过程特别有趣。

### 3. 百科知识

百科知识用于记录可脱离当前会话独立存在、并能在未来反复复用的客观知识。它强调的是“这是什么”“它通常意味着什么”“它有哪些背景资料”，而不是某个参与者的私人信息。这类记忆很多时候来自搜索、资料整理、作品设定、术语解释或梗知识总结。

二级分类：

- `文史`
- `理工`
- `生活`
- `娱乐`
- `梗知识`
- `其他`

示例：

- `娱乐`：《魔法少女的魔女审判》是由Re,AER旗下品牌Acacia制作的推理文字冒险游戏，中文发行由CIRCLE LINE GAMES负责。
- `梗知识`：哈基米最初源自日语“はちみつ”的空耳，后来演化成偏萌宠语境的网络梗。

### 4. 经验方法

经验方法用于记录从过去互动中总结出来、可在类似对象或类似情景下复用的策略、方法和理解方式。它强调的是“以后再遇到类似情况时怎样做更合适”，而不是“曾经发生了什么”或“这是什么知识”。

二级分类：

- `general`

示例：

- `general`：当对方突然发一个梗、表情包，很多时候不是在认真求解释，而是在借它表达情绪。更有效的方法是先理解语气和背后的情绪，再自然接话，而不是把注意力放在梗或表情包的字面内容上。

## 工具行为

1. 一次调用可以提交多个 query，参数为 `queries`，每个 query 都是一个独立检索单元，只负责一个明确的检索目标
2. 检索结果按查询分组返回，每组结果返回的是按相关性排序的候选提示 `hints`，但结果中仍可能存在噪声或弱相关项，需要自行鉴别和取舍
3. 某些 hint 可能带有 `msg_ids`，表示这条长期记忆关联的来源消息 ID，可用于后续追溯原始依据
4. 需要精确原文、消息时间线、图片或文件内容时，应使用技能 `query-chat-history-skill`

---

## 执行流程

1. 根据系统给出的检索要求，拆分本次需要检索的目标
2. 每个目标单独写成一个 query，不要把多个目标混在同一个 query 里
3. 为每个 query 选择合适的检索参数，并用自然语言描述要检索的长期记忆内容
4. 调用 `exec_skill` 执行 `search-long-term-memory-skill`

---

## Query 参数说明与检索指南

1. `memory`
   - 用自然语言描述要找什么，不要只堆关键词
   - 一个 query 只写一个检索目标；如果要查多个目标，就拆成多个 query
2. `index0`
   - 表示一级分类，必填
3. `index1`
   - 表示二级分类，可选
   - 宽泛检索时可以不指定，填空字符串 `""` 即可
4. `limit`
   - 表示返回数量上限，范围是 `10 ~ 50`
   - 宽泛检索、只指定一级分类时，建议直接使用 `50`
   - 同时指定一级分类和二级分类，但仍希望尽量多召回时，建议使用 `50`
   - 同时指定一级分类和二级分类，且目标较明确时，建议使用 `25`
   - 同时指定一级分类和二级分类，且检索目标已经非常精确时，可以使用 `10`

---

## 输入格式

```json
{
  "queries": [
    {
      "index0": "过往事件",
      "index1": "owner",
      "memory": "和 owner 有关的约定、承诺、答应过的事",
      "limit": 10
    },
    {
      "index0": "百科知识",
      "index1": "",
      "memory": "和某个作品有关的设定与背景资料",
      "limit": 50
    }
  ]
}
```

---

## 输出格式

```json
{
  "results": [
    {
      "query": {
        "index0": "过往事件",
        "index1": "owner",
        "memory": "和 owner 有关的约定、承诺、答应过的事",
        "limit": 10
      },
      "hints": [
        {
          "id": 12,
          "memory": "......",
          "createdAt": "2026-03-28 21:10:00",
          "msg_ids": [101, 105]
        }
      ]
    }
  ]
}
```

说明：

1. `results` 与输入 `queries` 一一对应，顺序保持一致
2. `query` 是该组 `hints` 对应的原始检索条件
3. `hints` 是命中的长期记忆候选提示，包含以下字段：
   - `id`：长期记忆的唯一标识符
   - `memory`：长期记忆内容
   - `createdAt`：长期记忆创建的时间
   - `msg_ids`：记忆相关的原始消息id列表，可能为空

---

## 结果使用注意事项

1. 检索结果按相关性排序返回，但仍可能包含噪声或弱相关项，需要自行鉴别和取舍
2. 如果命中多条相近结果时，要自行归并和取舍；如果命中结果彼此冲突，要根据实效性、相关性和可靠性进行判断
3. 需要严格证据时，通过 `query-chat-history-skill` 技能追溯原始消息内容
